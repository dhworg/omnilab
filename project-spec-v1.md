# Project Spec ‚Äî v1

**Working name:** OmniLab (rename later)
**Status:** v1 scope locked, architecture rev 2

---

## Pitch

A bootable, immutable Linux OS + CLI that gives ROS 2 + Gazebo developers a zero-setup, dependency-hell-proof environment for sim and hardware work. Beachhead: e-Yantra teams. Expansion: any robotics dev.

## Why

Robotics dev today loses hours to setup: wrong Ubuntu version, broken `rosdep`, RMW chaos, missing udev rules, USB serial that doesn't show up, GPU passthrough that won't. e-Yantra Stage 1 is purely simulation; even strong teams suffer because their environment fights them. The same pain hits every ROS 2 dev on a fresh laptop.

## Users

1. **Beachhead:** e-Yantra participants (~6000+/year)
2. **Expansion:** any ROS 2 / Gazebo developer

---

## Architecture

Three layers, kept deliberately separate.

### Layer 1 ‚Äî Host (`omnilab-host`)

A bootc-based immutable Linux image. Defined as a `Containerfile`, distributed as an OCI image, installed via an ISO generated by `bootc-image-builder`. Updates are atomic: `bootc upgrade` pulls a new image and reboots into it; rollback is one command.

The host contains *only*:
- Linux kernel + base userspace (Fedora bootc base, or Ubuntu bootc once stable)
- Display server (Wayland), KDE Plasma 6 desktop
- GPU drivers (Mesa for iGPU + NVIDIA proprietary; loads conditionally based on detected hardware)
- Container runtime (Podman) + nvidia-container-toolkit
- udev rules for the four target MCUs + USB cameras
- The `omnilab` CLI
- Default user with appropriate group memberships

The host does **not** contain ROS, Gazebo, or any project-specific tooling. That's the whole point.

### Layer 2 ‚Äî Project images (`omnilab-projects`)

OCI container images with pinned ROS 2 + Gazebo + extras. Each project's `omnilab.yaml` references an image by SHA256 digest, not tag. This is the actual dep-hell cure: identical bytes everywhere.

Default project images shipped:
- `ros-jazzy-base` ‚Äî minimal ROS 2 Jazzy
- `ros-jazzy-gz-harmonic` ‚Äî ROS 2 Jazzy + Gazebo Harmonic + ros-gz + nav2
- `eyrc-template` ‚Äî same as above, tuned for e-Yantra (low-res cameras, AR detection deps, common arenas)

Custom images: any user can write a `Containerfile` extending these.

### Layer 3 ‚Äî Skill-packs (`omnilab-skills`)

Optional, installable extensions. Each is a directory with manifest + install script + container layers. Pattern is forward-compatible: future vendor SDKs, RT kernel, cloud burst all slot in here without changing the core.

v1 ships one skill-pack:
- `llm-log-analyzer` ‚Äî Ollama + small model + service that watches Gazebo logs and ROS topics, suggests parameter tweaks. Disabled by default (RAM cost). One command to enable.

### CLI surface

```
omnilab new <name> [--template ros-jazzy-gz-harmonic]
omnilab up | down                    # start/stop project container
omnilab sim [--headless]             # launch Gazebo (GUI default)
omnilab perf-check <world.sdf>       # analyze SDF for perf issues
omnilab hw scan                      # enumerate connected MCUs
omnilab hw flash <board> <firmware>  # flash an MCU
omnilab micro-ros                    # start micro-ROS agent
omnilab skill install <name>
omnilab freeze                       # write lockfile from current state
omnilab doctor                       # diagnose host + project setup
```

### Manifest schema (`omnilab.yaml`)

```yaml
name: my-project
host_min_version: 0.1.0
image: ghcr.io/dhworg/ros-jazzy-gz-harmonic@sha256:<digest>
ros:
  rmw: rmw_cyclonedds_cpp
  domain_id: 42
gazebo:
  default_world: turtlebot3_world.sdf
  defaults:
    shadows: false
    camera_fps: 15
    camera_resolution: [320, 240]
gpu: auto              # auto | igpu | nvidia
hardware:
  micro_ros: enabled
  boards: [arduino_uno, esp32, stm32_blue_pill, rp2040]
skills: []
```

## v1 deliverable

Six artifacts:

1. `omnilab-host` bootc OCI image (published to GHCR)
2. Installable ISO + qcow2, generated from the host image
3. Three default project images (`ros-jazzy-base`, `ros-jazzy-gz-harmonic`, `eyrc-template`)
4. `omnilab` CLI (Python; ships in host)
5. `llm-log-analyzer` skill-pack (optional)
6. Documentation site (mkdocs) covering: install, hello-world, hardware quickstart, manifest reference, troubleshooting

USB ‚Üí fresh laptop ‚Üí working sim + hardware in 15 minutes, no terminal copy-paste.

## v1 must-do

1. Bootable ISO installs cleanly on commodity laptops, iGPU and NVIDIA both work
2. Host boots to KDE Plasma, default user has correct group memberships, udev rules active
3. `omnilab` CLI present, all listed commands functional
4. `omnilab new --template ros-jazzy-gz-harmonic` produces a working project in <60s
5. `omnilab sim` opens Gazebo with the demo world + RViz alongside; robot navigates
6. **Defaults that don't suck (GUI on by default, tuned for iGPU):**
   - Tutorial worlds: shadows off, single sun, no extra lights
   - Default camera resolution 320√ó240 @ 15Hz
   - RMW pinned to Cyclone DDS
   - Non-zero `ROS_DOMAIN_ID` default
   - Sim caps at 1.0 RTF when iterating
   - `--headless` flag is one keystroke away
7. **Hardware-ready out of the box:**
   - Groups: `dialout`, `plugdev`, `video`, `input` for default user
   - udev rules: FTDI, CP210x, CH340, ST-Link, RP2040 BOOTSEL
   - Toolchain in project image: `arduino-cli`, `esptool`, `dfu-util`, `stm32flash`, `picotool`, `picocom`, `screen`
   - PlatformIO preinstalled in project image
   - micro-ROS agent in project image, started by `omnilab micro-ros`
   - V4L2 utilities for USB cameras
8. **NVIDIA tier:** dGPU detected and used by Gazebo automatically; iGPU users see no difference
9. **`omnilab perf-check`:** parses an SDF, flags shadows, light count, camera resolution/rate, suggests tuning
10. Branding: custom `/etc/os-release`, boot splash, wallpaper

## Smoke tests

Six tests run on every ISO build (in CI) and on first boot (locally):

1. **Boot:** ISO installs in VM, host boots to KDE Plasma, login works
2. **CLI:** `omnilab doctor` returns all green
3. **Sim (headless, automated):** `omnilab sim --headless --test` runs nav2 to a goal, asserts goal reached, exits with PASS/FAIL
4. **Sim (GUI demo):** `omnilab sim` opens, GUI renders, robot moves on command (manual verification, scripted screenshot diff in CI)
5. **Hardware:** `omnilab hw scan` detects each of {Arduino Uno/Nano, ESP32, STM32 Blue Pill, RP2040}; `omnilab hw flash` succeeds for each; micro-ROS topic appears in `ros2 topic list`
6. **NVIDIA tier:** on a dGPU host, sim RTF >= 2√ó iGPU baseline for the demo world

## Development workflow

**Bootstrap (one-time):**

1. Push to `main` triggers GH Actions workflow `build-host-iso.yml`
2. Workflow builds the host OCI image, runs `bootc-image-builder` to produce ISO + qcow2, uploads as workflow artifacts
3. Download ISO, write to USB (`dd` or balenaEtcher), install on test machine
4. Test machine boots, gets a Tailscale IP, becomes Mac-reachable

**Daily loop ‚Äî Mac drives everything via SSH (Tailscale + tmux):**

The 80% case (truly live, no rebuild):
- `/etc/udev/rules.d/*` ‚Üí `udevadm control --reload && udevadm trigger`
- `/etc/systemd/system/*.service` ‚Üí `systemctl daemon-reload`
- `/etc/environment`, `/etc/profile.d/*.sh` (RMW pinning, `ROS_DOMAIN_ID`)
- `/etc/skel/*` (default user templates)
- Branding (`/etc/os-release`, GRUB themes, wallpapers)
- The `omnilab` CLI ‚Äî runs from `/var/home/parth/omnilab/` (a git checkout in mutable `/var`) during dev. Edit on Mac ‚Üí rsync ‚Üí run. Bake into the host image only at release time.

The 20% case (host image rebuild ‚Äî `/usr`, kernel modules, system packages):

Two paths depending on speed need:
- **Fast path (self-hosted, after bootstrap):** test machine builds its own host image with Podman.
  ```
  rsync Containerfile to test machine
  ssh: podman build -t localhost/omnilab-host:dev .
  ssh: bootc switch --transient localhost/omnilab-host:dev
  ssh: systemctl reboot
  ‚Üí 2‚Äì5 min cycle, rollback is `bootc rollback && reboot`
  ```
- **CI path (for shareable builds):** push to a feature branch, GH Actions builds, `bootc switch` to the GHCR-published image. ~10 min cycle but produces a real artifact and runs the full smoke-test matrix.

**Ephemeral probing:** `bootc usr-overlay` mounts a writable overlay on `/usr` for one-off "does this binary even work" tests. Reboot wipes it.

**CI matrix:**

`build-host-iso.yml` produces the ISO + qcow2 on every push to `main`, on PRs, and on tags.
`smoke-tests.yml` runs the six smoke tests against the qcow2 in a VM. NVIDIA + hardware tests run on a self-hosted runner (your dGPU machine, registered to GH; only runs on tags or label-triggered).

**Connectivity & ergonomics:**
- Tailscale between Mac and test machine ‚Äî stable hostnames through NAT
- `tmux` on test machine for long-running builds that survive SSH drops
- `mosh` instead of plain SSH if your home wifi is flaky

## Non-goals (v1)

Effort isn't the binding constraint anymore ‚Äî these are parked for *direction* or *external* reasons:

- **Headtracking / custom Wayland compositor** ‚Äî different project (was in original PDF, unrelated to dep hell)
- **From-scratch distro / LFS** ‚Äî different project
- **PREEMPT_RT real-time kernel** ‚Äî requires custom kernel build/test, RT tuning expertise; different engineering category
- **Vendor industrial-arm SDKs (UR, Franka, Robotiq)** ‚Äî licensing/distribution complexity; will be skill-packs when needed
- **Proprietary sensor SDKs (RealSense, ZED, Velodyne)** ‚Äî same; skill-pack pattern
- **Cloud GPU burst (`omnilab run --remote`)** ‚Äî needs cloud account + billing infra; skill-pack later
- **ARM / Raspberry Pi target** ‚Äî no test hardware on hand
- **Teensy support** ‚Äî no test hardware on hand
- **CAN bus tooling** ‚Äî specialized; skill-pack later
- **e-Yantra hardware-stage tooling** (specific vendor kits) ‚Äî skill-pack territory

## First steps

With Claude Code in the loop these can be parallelized; ordered by dependency.

**Phase A ‚Äî Bootstrap (the pipeline must work end-to-end before anything else gets built on top):**

1. **Repo scaffold + GH Actions `build-host-iso.yml`.** Directory structure, mkdocs skeleton, README, and a working CI workflow that produces a *minimal* host ISO. The host image at this point is just Fedora bootc + XFCE + Podman + a `hello-omnilab` script. No CLI yet, no skill-packs. The point is to prove the build-and-publish pipeline works.
2. **Test machine bootstrap.** Download the ISO from the GH Actions artifact, USB-flash, install on the dGPU machine. Set up Tailscale, tmux, SSH access from Mac. Run `bootc switch` against a manually-pushed image variant to confirm the live-update loop works. This is the milestone where Mac-as-cockpit becomes real.

**Phase B ‚Äî Build out the layers (parallelizable after Phase A):**

3. **Project image: `ros-jazzy-gz-harmonic`.** Containerfile with ROS 2 Jazzy + Gazebo Harmonic + ros-gz + PlatformIO + firmware tools + micro-ROS agent. Test standalone with `podman run` on the test machine.
4. **CLI v0.** Python package implementing `new`, `up`, `down`, `sim`, `hw scan`, `doctor`. Calls `podman` under the hood. During dev, lives in `/var/home/parth/omnilab/` (mutable, fast iteration).
5. **Host image ‚Äî full version.** Add udev rules, group setup, nvidia-container-toolkit, branding, the CLI baked in. Replace the minimal Phase-A host.
6. **Smoke tests.** All six, scripted, runnable in CI and on first boot via `omnilab doctor --full`. Add `smoke-tests.yml` workflow.

**Phase C ‚Äî Verification (gates v1 release, can't be parallelized):**

7. **NVIDIA tier verification.** Boot ISO on dGPU machine, run smoke test 6.
8. **Hardware verification.** Plug in each of the four MCUs, run smoke test 5 end-to-end.

**Phase D ‚Äî Polish:**

9. **`perf-check` + `llm-log-analyzer` skill-pack.** Smaller scope, parallel with Phase C.
10. **Docs site.** Quickstart, hardware setup, manifest reference, troubleshooting.
